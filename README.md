# Миссии Николая

«Миссии Николая» — семейная PWA с двумя отдельными интерфейсами и общей облачной базой. Николай отправляет выполненную миссию, родитель подтверждает или отклоняет её, а XP и доступные минуты обновляются на обоих устройствах.

## Первый запуск семьи

1. Откройте **ссылку родителя**.
2. Введите email и откройте одноразовую magic-link из письма на этом же устройстве.
3. Укажите имя ребёнка «Николай» и задайте PIN из четырёх цифр.
4. После входа по PIN откройте карточку **«Одноразовое приглашение»**.
5. Создайте ссылку и откройте её на телефоне Николая или отсканируйте локально сгенерированный QR-код.
6. На телефоне Николая подтвердите имя. Одноразовый token исчезнет из истории адреса, а устройство получит ограниченную детскую сессию.

Родительская и детская сессии должны находиться на разных устройствах или в разных профилях браузера. Вкладки одного профиля делят хранилище сайта и поэтому не считаются независимыми устройствами.

## Что умеют интерфейсы

Детский интерфейс показывает только миссии, обязательный минимум, статусы заявок, XP, доступные минуты, уровень, прогресс и таймер. Ребёнок может отправить миссию, выбрать количество отжиманий, запустить и остановить доступный таймер. Административной навигации в детском маршруте нет.

Родительский интерфейс включает:

- email-аутентификацию и локальную PIN-блокировку;
- очередь заявок, подтверждение, отклонение и отмену начисления;
- настройку XP, формулы минут, дневного лимита, переноса остатка и темы;
- управление миссиями, ручную корректировку XP, историю и статистику;
- состояние таймера ребёнка;
- ссылку родителя, одноразовую ссылку Николая и QR-код;
- отзыв детской сессии и открытых приглашений;
- выход из облачной учётной записи и удаление семейных данных;
- автоматическую блокировку после пяти минут бездействия, сворачивания или ухода с экрана.

## PIN и восстановление доступа

PIN не является серверным паролем. Основная личность родителя подтверждается Supabase Auth через email; PIN — дополнительный локальный барьер на уже авторизованном устройстве.

- Смена PIN: **Родителю → Настройки → Сменить PIN**.
- Если PIN забыт: на экране блокировки нажмите **«Забыли PIN?»** и задайте новый. Команда доступна только в действующей родительской email-сессии.
- PIN никогда не передаётся и не хранится открытым текстом. В базе находится salted SHA-256 hash и отдельная случайная соль, недоступные детской роли.
- Если потерян доступ к email, восстановление выполняется у почтового провайдера; PIN не заменяет email-аутентификацию.

## Переподключение телефона Николая

В родительском интерфейсе нажмите **«Отключить детское устройство»**. Сервер немедленно отзывает активное детское membership и все открытые приглашения. Затем создайте новую одноразовую ссылку и откройте её на нужном устройстве. У семьи одновременно только одна активная детская сессия; подключение новой отзывает прежнюю.

## Синхронизация и offline

Supabase — источник истины. Realtime сообщает клиентам об изменениях, а резервный polling выполняется каждые 30 секунд, при возврате во вкладку и после восстановления сети.

- Снимок семьи кэшируется локально для чтения после первой online-загрузки.
- Детская отправка миссии offline попадает в устойчивую FIFO-очередь с постоянным operation ID и безопасно повторяется после reconnect.
- Родительские изменения требуют сети. При ошибке операция не изображается успешной; повтор после потерянного ответа использует прежний idempotency key.
- App shell, manifest, иконки и последняя production-версия кэшируются service worker.
- Старый localStorage не загружается в облако автоматически. Автономная версия остаётся по адресу `#/local` как отдельный демо-режим.

## Установка PWA

### iPhone и iPad

1. Откройте нужную production-ссылку в Safari.
2. Нажмите **«Поделиться» → «На экран Домой» → «Добавить»**.
3. Запускайте приложение с установленной иконки. Не используйте приватную вкладку или встроенный браузер мессенджера.

### Android

1. Откройте нужную ссылку в Chrome.
2. Выберите в меню **«Установить приложение»** или **«Добавить на главный экран»**.
3. Подтвердите установку. Первая загрузка требует сети.

## Облачная архитектура

Frontend: React 19, TypeScript, Vite 7, Web App Manifest и service worker. Hosting: GitHub Pages. Backend: Supabase Postgres, Auth и Realtime.

Основные таблицы:

- `families`, `family_members`, `family_settings`, `parent_secrets`;
- `tasks`, `daily_family_states`, `daily_task_states`;
- append-only `reward_transactions`;
- `timer_sessions`, `history_days`, `activity_events`;
- digest-only `family_invites` и `rpc_operations` для идемпотентности.

Все browser-записи проходят через `SECURITY DEFINER` RPC. Сервер берёт пользователя из `auth.uid()`, сам определяет family/role, блокирует конфликтующие строки и ведёт revision. Клиент не передаёт доверенную роль, family ID или размер награды.

Миграции находятся в `supabase/migrations/` и применяются по порядку:

1. схема, ограничения и append-only ledger;
2. авторизация, rollover, snapshot и приглашения;
3. детские/родительские действия и таймер;
4. RLS, grants и Realtime publication;
5. release hardening Auth, семейных блокировок и единственного открытого invite;
6. отзыв детского устройства.

`supabase/tests/database/` содержит 25 транзакционных pgTAP-проверок схемы, RLS, grants, функций, immutable ledger и отсутствия invite plaintext.

## Безопасность

- Во frontend используется только публичный Supabase publishable key.
- Service-role key, database password и management token не входят в Git, Actions Variables или bundle.
- RLS включён на каждой облачной таблице; прямые INSERT/UPDATE/DELETE для browser roles закрыты.
- Ребёнок не может подтвердить задание, назначить награду, изменить настройки/PIN, удалить ledger или стать родителем.
- Родительская семья создаётся только неанонимной email-сессией.
- Invite действует 5–1440 минут, используется один раз, а в БД хранится только SHA-256 digest.
- Один approval revision создаёт максимум одну награду; повтор ответа/клика воспроизводит сохранённый результат.
- Ledger append-only; отмена создаёт отдельную reversal-транзакцию.
- Release scan блокирует private keys, GitHub/Supabase server tokens, credential URLs и localhost endpoints в production bundle.

## Удаление и резервное копирование

Удаление доступно в родительском интерфейсе после ввода точного названия семьи. В одной транзакции удаляются семейные настройки, миссии, статусы, история, начисления, таймеры, приглашения и memberships. Supabase Auth identity/email не удаляется автоматически.

У приложения пока нет пользовательского cloud-export/import. Локальный demo имеет резервную запись только в том же браузере. Для операторской резервной копии владелец проекта может использовать штатные возможности Supabase Dashboard/CLI; доступность автоматических backup зависит от текущего тарифа Supabase. Перед удалением семьи считайте действие необратимым.

## Локальный запуск

Требуется Node.js 24 LTS.

```bash
git clone https://github.com/kalish1213-eng/missions-nikolay.git
cd missions-nikolay
npm ci
copy .env.example .env.local
npm run dev
```

Публичные переменные:

```dotenv
VITE_SUPABASE_URL=https://PROJECT_REF.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=sb_publishable_...
```

Не добавляйте `service_role`, `sb_secret_...`, database password или Supabase access token в переменные `VITE_*`.

Проверки:

```bash
npm run lint
npm run typecheck
npm test
npm run build
node scripts/scan-release.mjs --source .
node scripts/scan-release.mjs --release dist
```

## GitHub Pages и повторный deploy

Workflow `.github/workflows/deploy-pages.yml` запускается при pull request, push в `main` и вручную. Он выполняет `npm ci`, проверку публичной конфигурации, secret scan, lint, typecheck, 100 тестов, production build, release scan и только затем официальный Pages deploy.

В repository variables должны существовать:

- `VITE_SUPABASE_URL`;
- `VITE_SUPABASE_PUBLISHABLE_KEY`.

Повторный deploy запускается новым push в `main` или через **Actions → Deploy PWA to GitHub Pages → Run workflow**. `base: './'` сохраняет работу assets, manifest и service worker в подпапке GitHub Pages.

## Известные ограничения

1. PWA считает экранное время, но физически не блокирует телефон; используйте Screen Time/Family Link для системного ограничения.
2. ОС может заморозить фоновое PWA. После возврата серверный таймер пересчитывается по реальному времени, но точный фоновый звук не гарантируется.
3. Offline поддерживает чтение кэша и очередь детской отправки; родительские изменения выполняются online.
4. Одновременно поддерживается одна активная детская сессия на семью.
5. Облачный импорт старого localStorage намеренно не автоматизирован, чтобы не дублировать начисления.
6. Браузерная проверка двух контекстов не заменяет приёмку на физических iPhone/Android.

Подробности предыдущей мобильной визуальной приёмки: [`qa/QA_REPORT.md`](qa/QA_REPORT.md).
